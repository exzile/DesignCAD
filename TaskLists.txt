## ARACHNE-8: complete the variable-width-wall implementation
- [ ] Current state (this codebase, 2026-04-25): the Arachne pipeline at
      `src/engine/slicer/pipeline/arachne/` runs Voronoi → trapezoidation →
      bead distribution → path extraction and passes its 23 unit + 16
      integration tests on synthetic fixtures. Default is classic until
      this is finished because two real-world failure modes remain:
        (1) Path extraction at degree-3+ Voronoi vertices: my fragments
            don't merge across branches, so on dense polygons most beads
            are emitted as standalone short paths. Codex's `branchKeys`
            filter blocks merging at junctions to avoid chord-crossing,
            but this leaves walls fragmented. Real Arachne walks a half-
            edge graph topologically to determine which 2 of N junction
            fragments belong to the same wall ring.
        (2) No transition zones: when adjacent trapezoids have different
            bead counts (e.g. 3-bead wide region next to 2-bead narrow
            neck), the "extra" bead in the wider trapezoid abruptly ends
            at the junction, producing visible end-cap parallelograms in
            breakthrough/neck regions (e.g. between mounting holes).
            Cura/Orca's Arachne tapers the dying bead's width to zero
            over a transition length (~3 × lineWidth) for smooth output.
        Effect on adjustable_support_foot: layer 21 has 1700-2500 polygon
        edges per region, hits the 400-edge cap, falls back to classic.
        Even if the cap were raised, the path-extraction limitation would
        produce poor wall topology there.
      Three viable paths to a real fix:
        (a) Implement transition zones + graph-traversal path extraction.
            ~3-5 days of focused algorithm work + tests. Pure-JS, no new
            deps. Reference: Cura's `SkeletalTrapezoidation::generate*`
            and `BeadingStrategy/DistributedBeadingStrategy.cpp` for the
            transition logic (libArachne).
        (b) WASM build of `boost::polygon::voronoi` + Cura's libArachne.
            Sidesteps both limitations. ~1-2 days for the Emscripten
            build setup; the existing TS adapter (`buildEdgeVoronoi`'s
            return type) is already shaped to receive its output.
        (c) Pre-simplify polygon contours upstream of `generatePerimeters`
            using an AREA-PRESERVING method (morphological closing via
            `offsetContour(+ε)` then `offsetContour(-ε)` is the canonical
            JS approach — polygon-clipping doesn't have a built-in
            simplifier, and RDP shrinks area). Doesn't fix Arachne, but
            reduces tessellation noise so the cap stops triggering and
            classic produces cleaner output too. ~1 day.
      Recommended order: (c) first as a quick win (helps both classic and
      Arachne), then (b) for the proper Arachne. (a) is a fallback if the
      WASM build proves too painful.

## ARACHNE-9 PHASE 1 — Toolchain + boost::polygon::voronoi
- [ ] **9.1A** [owner: claude] Set up Emscripten via Docker so the build is
      reproducible without polluting the dev's local toolchain. Deliverable:
      `wasm/Dockerfile` that pulls `emscripten/emsdk:3.1.x`, vendors Boost
      headers (just the `boost.polygon` and `boost.geometry` headers — no
      compiled libs needed), and `wasm/build.sh` that runs `em++` to
      produce `wasm/dist/voronoi.{js,wasm,d.ts}`. Validate by building a
      hello-world `voronoi.cpp` that returns `1.0` from a single function.
      ~1 day.

- [ ] **9.1B** [owner: codex] In parallel with 9.1A, set up Vite-side WASM
      loading. Add `vite-plugin-wasm` (or similar) to `vite.config.ts` with
      a glob pattern that includes `**/*.wasm`, configure asset handling so
      the `.wasm` artifact gets a content-hash filename + lazy-load behavior
      (NOT bundled into the worker's initial chunk). Wire a `wasm-test/`
      module that loads any small `.wasm` file and asserts it returns the
      expected value. Doesn't depend on the actual libArachne build — uses
      a stub `.wasm` so this can ship before 9.1A is done.
      ~0.5 day.

- [ ] **9.1C** [owner: claude, depends on 9.1A] Write `wasm/src/voronoi.cpp`
      — a thin C++ wrapper around `boost::polygon::voronoi_diagram` that
      takes a flat array of segment endpoints (`double[4]` per segment),
      runs Boost's Fortune sweep-line, and writes results back to a
      preallocated output buffer in a known layout (vertex count + flat
      `double[3]` per vertex (x, y, radius), edge count + `int[4]` per
      edge (fromVertex, toVertex, sourceEdgeA, sourceEdgeB)). NO `embind`
      — use raw C exports + manual `Module.HEAPF64` buffer marshaling on
      the JS side; this minimises bundle size and keeps the binding layer
      auditable. Add `wasm/test/voronoi.test.cpp` that runs the existing
      `voronoi.test.ts` fixtures (rectangle, lShape, hexagon, annulus,
      thinNeck, breakthroughHole) through the C++ shim and verifies the
      same vertex+edge counts.
      ~1 day.

- [ ] **9.1D** [owner: claude, depends on 9.1C] Write the TS adapter
      `src/engine/slicer/pipeline/arachne/voronoiWasm.ts` that exposes the
      same `buildEdgeVoronoi(outerContour, holeContours): VoronoiGraph`
      signature as the existing pure-JS `voronoi.ts`. Implementation:
      flatten contours to a `Float64Array`, copy into the WASM module's
      heap, call `_buildVoronoi`, copy results back, reconstruct
      `VoronoiVertex` / `VoronoiEdge` objects. Key constraint: the public
      type signature MUST match `voronoi.ts` exactly so the rest of the
      pipeline doesn't change.
      ~0.5 day.

- [ ] **9.1E** [owner: claude, depends on 9.1D + 9.1B] Run the existing 23
      unit tests in `voronoi.test.ts` against the WASM adapter (parameterise
      the test file with both implementations). All 23 must pass. Compare
      output on the user's `adjustable_support_foot` layer 21 (1700+ edge
      polygon) — WASM Voronoi should complete in <100ms instead of the
      current minutes-per-layer.
      ~0.5 day.
