# AI ASSISTANT — MCP server + optional in-app chat  (architectural, multi-session)
# ============================================================================
# Goal: let users drive DesignCAD with their own Claude / Codex / OpenAI
# subscription. Open-source-friendly: no Anthropic API spend on our side,
# no shared key. Two consumption paths:
#   (a) MCP server inside DesignCAD that exposes CAD tools. The user runs
#       Claude Code locally and adds DesignCAD as an MCP server — chat
#       happens in their terminal/IDE, geometry shows up in the running
#       browser session. Subscription quota covers it because Claude Code
#       is the authenticated client.
#   (b) BYOK chat panel inside DesignCAD as a fallback for users who don't
#       run Claude Code. The same MCP tool surface is reused as the
#       function-call schema for the BYOK path so we don't ship two APIs.
#
# Rule of the road: the MCP server is the source of truth. Every CAD
# action the assistant can take must exist as an MCP tool first; the chat
# panel is just a thin client over that surface.

## Phase 1 — MCP server scaffolding
#   New dir: src/services/mcp/
#   [ ] Add @modelcontextprotocol/sdk dependency + lock CORS to localhost
#       origins (the server only ever talks to the user's own Claude Code)
#   [ ] Implement an HTTP + SSE transport bound to a configurable port
#       (default 5174; the Vite dev server is on 5173, so the two coexist)
#   [ ] Pairing token: generate a one-time token, surface it in the new
#       AI Assistant settings tab as a copy-paste line
#       (`claude mcp add designcad http://localhost:5174/mcp?token=...`).
#       Server rejects requests without it.
#   [ ] Lifecycle: server runs only while the DesignCAD tab is open;
#       reconnects survive HMR; `stop()` on tab close
#   [ ] Codex-friendly: this whole phase is plumbing — no domain
#       knowledge required.

## Phase 2 — Core tool surface
#   New dir: src/services/mcp/tools/
#   Each tool is a thin wrapper over an existing store action so the
#   assistant can't reach into anything the UI itself can't do. Tools
#   are typed with zod schemas and registered via a single index.ts.
#
#   Document / scene
#   [ ] list_objects                  — returns id, kind, bbox, transform
#   [ ] get_object_properties         — full feature/sketch tree for one id
#   [ ] select_objects                — drives the same selection store
#   [ ] snapshot_view                 — returns a PNG of the viewport (use
#                                       the existing exporter; the result is
#                                       returned as an MCP "image" content
#                                       item so the model can SEE its work)
#
#   Primitives
#   [ ] create_box        { x, y, z, position?, rotation? }
#   [ ] create_cylinder   { radius, height, position?, axis? }
#   [ ] create_sphere     { radius, position? }
#   [ ] create_cone       { r1, r2, height, position? }
#
#   Sketches
#   [ ] start_sketch      { plane: 'XY'|'XZ'|'YZ' | object id+face }
#   [ ] sketch_rect       { x, y, w, h, centered? }
#   [ ] sketch_circle     { cx, cy, radius }
#   [ ] sketch_polygon    { points: [[x,y],...] }
#   [ ] sketch_dimension  { entityId, value }
#   [ ] finish_sketch
#
#   Features
#   [ ] extrude_sketch    { sketchId, depth, direction? }
#   [ ] revolve_sketch    { sketchId, axis, angle? }
#   [ ] fillet_edges      { objectId, edgeIds[], radius }
#   [ ] chamfer_edges     { objectId, edgeIds[], distance }
#   [ ] hole              { face, position, diameter, depth, throughAll? }
#
#   Booleans + transforms
#   [ ] boolean_union          { idsA[], idsB[] }
#   [ ] boolean_subtract       { from, tools[] }
#   [ ] boolean_intersect      { ids[] }
#   [ ] transform              { id, translate?, rotateDeg?, scale? }
#   [ ] mirror                 { id, plane }
#   [ ] linear_pattern         { id, axis, count, spacing }
#   [ ] circular_pattern       { id, axis, count, totalAngle }
#
#   Export / save
#   [ ] save_session
#   [ ] export_stl    { ids?: string[] }
#   [ ] export_step   { ids?: string[] }
#   [ ] export_gcode  { profileId? }
#
#   Each tool's handler returns a structured result the model can read
#   ({ created: id, bbox, ... }) so multi-step plans (extrude → fillet)
#   work without hallucinated ids.
